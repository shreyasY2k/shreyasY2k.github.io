Branch Strategy

A well-structured branch strategy is essential for team collaboration. Our recommended approach uses protected branches for production and staging environments, with feature branches for development.

Branch Structure Visualization

Branch Types and Their Purpose:

  • Production Branch (protected): Contains code that is currently deployed to the production environment. Direct pushes are not allowed - changes must go through pull requests with appropriate reviews.
  • Staging Branch (protected): Used for pre-release testing before code goes to production. This branch should mirror what will be deployed to production.
  • Develop Branch: Main integration branch where features are merged for development testing. This is where features come together before being promoted to staging.
  • Feature Branches: Created from develop branch, where individual features are built. Each feature gets its own branch (e.g., feature/login, feature/dashboard).
  • Hotfix Branches: Created from production branch for urgent fixes. After fixing the issue, changes are merged to production, staging, and develop branches.

Branch Protection Rules

To maintain code quality and prevent accidental changes, configure these protection rules in GitHub:

For Production Branch:

  • Require pull request reviews before merging
  • Require approval from at least 2 reviewers
  • Require status checks to pass (CI/CD tests)
  • Restrict who can push to matching branches
  • Disable force pushes
  • Require linear history

For Staging Branch:

  • Require pull request reviews before merging
  • Require approval from at least 1 reviewer
  • Require status checks to pass
  • Disable force pushes

Branch Strategy Quiz

Test your understanding of branch strategy by answering these questions:

1. Where should feature branches be created from?

Correct! Feature branches should always be created from the develop branch, which is the main integration branch for development.

2. When a critical bug is found in production, what kind of branch should you create?

Correct! Hotfix branches are created directly from the production branch to fix critical issues. This allows you to make emergency fixes without including unrelated features that might be in develop.

3. Which branches should be protected in GitHub?

Correct! Production and staging branches should be protected to ensure code quality and prevent accidental pushes. The develop branch may also be protected in some organizations, but feature branches typically don't need protection.

4. After merging a hotfix to production, what should you do next?

Correct! After fixing a production issue, you need to propagate that fix to staging and develop branches to ensure all branches include the fix in future deployments.

Merge Conflict Resolution

Merge conflicts occur when two branches have made different changes to the same line of code. Learning how to resolve these conflicts effectively is a critical skill for team collaboration.

Understanding Merge Conflicts

What Causes Conflicts?

  • Two branches modifying the same line of code
  • One branch deleting a file while another modifies it
  • Both branches adding files with the same name but different content
  • Incompatible changes to related code sections

Conflict Markers

<<<<<<< HEAD
Your changes on current branch
=======
Incoming changes from other branch
>>>>>>> branch-name

Git inserts these markers to show you both versions of the conflicting code.

Visual Example

Your Branch (HEAD)
function calculateTotal(items) {
  return items.reduce((sum, item) => 
    sum + item.price * item.quantity, 0
  );
}
Their Branch (develop)
function calculateTotal(items) {
  return items
    .filter(item => item.inStock)
    .reduce((sum, item) => sum + item.price, 0);
}
Conflict in Your Editor
<<<<<<< HEAD
  return items.reduce((sum, item) => 
    sum + item.price * item.quantity, 0
  );
=======
  return items
    .filter(item => item.inStock)
    .reduce((sum, item) => sum + item.price, 0);
>>>>>>> develop
Resolved Version (Combined Features)
function calculateTotal(items) {
  return items
    .filter(item => item.inStock)
    .reduce((sum, item) => 
      sum + item.price * item.quantity, 0
    );
}

Conflict Resolution Steps:

1

Update your local develop branch

user@laptop:~/project$ git checkout develop
Switched to branch 'develop'
user@laptop:~/project$ git pull origin develop
Already up to date.

Always start with the latest version of develop to ensure you're working with current code.

2

Switch to your feature branch

user@laptop:~/project$ git checkout feature/payment
Switched to branch 'feature/payment'
3

Merge develop into your feature branch

user@laptop:~/project$ git merge develop
Auto-merging src/utils/payments.js
CONFLICT (content): Merge conflict in src/utils/payments.js
Automatic merge failed; fix conflicts and then commit the result.

Git will automatically merge what it can and mark conflicts it can't resolve automatically.

4

Check status to see conflicting files

user@laptop:~/project$ git status
On branch feature/payment
You have unmerged paths.
(fix conflicts and run "git commit")
(use "git merge --abort" to abort the merge)
Unmerged paths:
(use "git add ..." to mark resolution)
both modified: src/utils/payments.js

Tip: If you want to abort the merge and try a different approach, you can use:
git merge --abort

5

Open conflicting files in your editor

You'll see conflict markers showing both versions of the code:

function calculateTotal(items) { <<<<<<< HEAD return items.reduce((sum, item)=> sum + item.price * item.quantity, 0 ); ======= return items .filter(item => item.inStock) .reduce((sum, item) => sum + item.price, 0); >>>>>>> develop }
6

Edit the file to resolve conflicts

Modify the file to include the best parts of both changes, and remove the conflict markers:

function calculateTotal(items) { return items .filter(item => item.inStock) .reduce((sum, item) => sum + item.price * item.quantity, 0 ); }

Important: Make sure to remove ALL conflict markers (<<<, =====, >>>) from the file. The file must not contain any of these markers after resolution.

7

Mark as resolved and complete the merge

user@laptop:~/project$ git add src/utils/payments.js
user@laptop:~/project$ git status
On branch feature/payment
All conflicts fixed but you are still merging.
(use "git commit" to conclude merge)
user@laptop:~/project$ git commit -m "Merge develop and resolve conflicts in payment calculation"
[feature/payment abc1234] Merge develop and resolve conflicts in payment calculation
8

Push your resolved branch

user@laptop:~/project$ git push origin feature/payment

Best Practices for Conflict Resolution

Prevention

  • Pull from develop regularly to keep your branch updated
  • Coordinate with team members working on similar areas
  • Keep feature branches short-lived and focused
  • Follow modular code design to minimize overlapping changes

Resolution Tips

  • Understand both changes before resolving conflicts
  • Consult with the developer who made the other changes
  • Use visual diff tools like VS Code's merge editor
  • Test your code after resolving conflicts
  • Commit descriptive merge messages

Using Visual Merge Tools

Most modern code editors have built-in merge conflict resolution tools:

  • VS Code: Opens files with conflicts in a special merge editor with "Accept Current", "Accept Incoming", and "Accept Both" options
  • IntelliJ/WebStorm: Provides a visual three-panel merge tool
  • Git GUI tools like GitKraken and Sourcetree have visual conflict resolution

You can configure your preferred merge tool with:
git config --global merge.tool [your-tool-name]

Hotfix Process

When a critical bug is discovered in production, you need a process to fix it quickly while minimizing risk. The hotfix workflow provides a structured approach to handling these urgent situations.

Hotfix Flow Visualization

What Makes a Good Hotfix?

Do:

  • Focus only on fixing the specific issue
  • Write thorough tests to verify the fix
  • Document what caused the issue
  • Keep the changes as minimal as possible
  • Consider performance impacts in production

Don't:

  • Include unrelated code improvements
  • Refactor surrounding code
  • Fix other minor bugs you notice
  • Add new features
  • Change more files than necessary

Hotfix Process Steps:

1

Start from production branch

user@laptop:~/project$ git checkout production
Switched to branch 'production'
user@laptop:~/project$ git pull origin production
Already up to date.

Always start with the latest version of production to fix the issue as it appears in the live environment.

2

Create a hotfix branch

user@laptop:~/project$ git checkout -b hotfix/auth-timeout
Switched to a new branch 'hotfix/auth-timeout'

Use a descriptive name with the prefix "hotfix/" that clearly indicates what you're fixing.

3

Fix the issue with minimal changes

user@laptop:~/project$ git status
On branch hotfix/auth-timeout
nothing to commit, working tree clean
user@laptop:~/project$ # Make your fix
user@laptop:~/project$ git add src/services/auth.js
user@laptop:~/project$ git commit -m "Fix authentication timeout issue (#253)"
[hotfix/auth-timeout abc1234] Fix authentication timeout issue (#253)
1 file changed, 3 insertions(+), 1 deletion(-)

Important Best Practice:

Reference the issue number in your commit message to create an automatic link in GitHub. This helps with tracking and creates a record of what the hotfix addresses.

4

Test thoroughly

Run all relevant unit tests and manually test the fix to ensure it resolves the issue without introducing new problems.

user@laptop:~/project$ npm run test:auth
✓ Auth Service tests completed successfully
✓ Authentication timeout test passed
All tests passed!
5

Push your branch and create the first PR to production

user@laptop:~/project$ git push origin hotfix/auth-timeout
Enumerating objects: 5, done.
Counting objects: 100% (5/5), done.
Delta compression using up to 8 threads
Compressing objects: 100% (3/3), done.
Writing objects: 100% (3/3), 320 bytes | 320.00 KiB/s, done.
Total 3 (delta 2), reused 0 (delta 0)
remote: Resolving deltas: 100% (2/2), completed with 2 local objects.
To github.com:company/project.git
* [new branch] hotfix/auth-timeout -> hotfix/auth-timeout

Create a Pull Request targeting the production branch first.

Production PR Title:

HOTFIX: Fix authentication timeout issue (#253)

Description:

This hotfix addresses the critical authentication timeout issue affecting production:

  • Increases the token expiration time from 15 to 30 minutes
  • Adds proper error handling for timeout scenarios
  • Updates error message to be more user-friendly

This issue was causing users to be logged out during long form submissions.

Root Cause: The timeout was set too low for complex operations.

Testing: Verified with automated tests and manual testing of the submission flow.

Fixes #253

6

After the production PR is approved and merged

Create additional Pull Requests to merge the hotfix to:

  1. Staging branch - to keep staging in sync with production
  2. Develop branch - to ensure the fix is included in future releases

Important: The order matters! Always merge to production first, then to staging, and finally to develop. This ensures that the hotfix is properly propagated across all environments.

7

Deploy to production

After the hotfix is merged to production, deploy it immediately following your standard production deployment process.

user@laptop:~/project$ npm run deploy:production
Deploying to production...
Building assets...
Running tests...
Uploading build artifacts...
Deployment successful!
8

Monitor the fix in production

Watch logs, error rates, and key metrics closely after deploying to ensure the fix resolves the issue without causing any new problems.

Post-Hotfix: Create a follow-up task to address the root cause more thoroughly in the develop branch if needed. Sometimes hotfixes are minimal by necessity, and a more comprehensive solution can be implemented for the next release.

Team Collaboration

When multiple developers work on the same feature or codebase, effective coordination is essential. This section covers strategies for smooth collaboration while minimizing conflicts and maintaining code quality.

Team Collaboration Approaches

Shared Feature Branch

  • Everyone works directly on the same branch
  • Simple setup, direct collaboration
  • Regular communication essential
  • Pull before starting work each day
  • Higher potential for conflicts

Task Branches

  • Each developer creates personal task branches
  • Task branches merge to the feature branch
  • Better isolation and organization
  • Task-specific code reviews
  • More branches but fewer direct conflicts

Collaboration Best Practices:

Before Starting Work

  • Discuss the feature implementation with your team
  • Break down the feature into smaller tasks
  • Decide on which files each person will modify
  • Pull the latest changes before starting
  • Agree on coding standards and patterns

During Development

  • Commit frequently with focused changes
  • Pull and integrate changes regularly
  • Communicate when working on shared files
  • Keep the team updated on your progress
  • Request early feedback on complex solutions

When Submitting Work

  • Pull the latest changes before pushing
  • Ensure all tests pass before creating a PR
  • Write detailed PR descriptions
  • Request reviews from relevant team members
  • Address review comments promptly

Conflict Prevention Strategies

Code Organization

  • Use modular code architecture
  • Separate concerns into different files
  • Minimize dependencies between components
  • Follow consistent coding patterns
  • Keep files focused on single responsibilities

Communication

  • Hold daily stand-ups or check-ins
  • Use branch naming to indicate who's working on what
  • Comment on issues/tasks with status updates
  • Use pull requests for early feedback
  • Document architectural decisions

Pro Tip: Create a CODEOWNERS file in your repository to automatically assign reviewers based on file paths: 

# Frontend components
/src/components/ @frontend-team

# Backend services
/src/services/ @backend-team

# Authentication files
/src/auth/ @security-team @backend-team

Common Collaboration Scenarios

Multiple Developers on Same Feature

Scenario:

Three developers need to implement a new user dashboard with charts, filters, and data tables.

Approach:
  • Create a shared feature/dashboard branch
  • Divide responsibilities by component:
    • Dev 1: Chart components
    • Dev 2: Filter components
    • Dev 3: Data table components
  • Set up daily check-ins to coordinate integration
  • Create shared interfaces/contracts first
Terminal:
dev1@laptop:~/project$ git checkout develop
dev1@laptop:~/project$ git pull origin develop
dev1@laptop:~/project$ git checkout -b feature/dashboard
dev1@laptop:~/project$ git push -u origin feature/dashboard
dev1@laptop:~/project$ # Create shared interfaces
dev1@laptop:~/project$ git add src/interfaces/dashboard.ts
dev1@laptop:~/project$ git commit -m "Add dashboard interfaces"
dev1@laptop:~/project$ git push
Success Factors:
  • Clear component boundaries
  • Agreement on data structures
  • Regular communication
  • Frequent small pushes
  • Shared understanding of acceptance criteria
Common Pitfalls:
  • Working on the same files without coordination
  • Not pulling frequently enough
  • Making large, sweeping changes
  • Pushing broken code to the shared branch
  • Not following agreed-upon patterns and interfaces

Breaking Down a Complex Feature

Scenario:

Building a complete payment processing system with frontend, API, database changes, and third-party integrations.

Approach:
  • Create a main feature/payments branch
  • Create sub-feature branches:
    • feature/payments-ui
    • feature/payments-api
    • feature/payments-integration
  • Merge sub-features into the main feature branch
  • Test integration on the main feature branch
  • Create final PR from feature/payments to develop
Branch Flow:
develop
  └── feature/payments
       ├── feature/payments-ui
       ├── feature/payments-api
       └── feature/payments-integration
Benefits:
  • Allows parallel development
  • Easier to review smaller components
  • Integration testing before merging to develop
  • Clear responsibility for each part
  • Lower risk of breaking changes
Pro Tip:

Use feature flags to integrate incomplete features into develop without exposing them to users. This allows for earlier integration and testing.

Collaborative Refactoring

Scenario:

Your team needs to refactor a core part of the application that many developers rely on.

Approach:
  • Create a refactor/core-service branch
  • Clearly document the plan and interfaces
  • Set aside dedicated time for team refactoring
  • Use pair programming for critical sections
  • Maintain backward compatibility where possible
  • Write thorough tests before and after changes
Communication is Key:

For major refactoring, consider a brief code freeze or coordinated work session where the entire team focuses on the refactoring together.

Refactoring Strategies:
  1. Strangler Pattern: Gradually replace old functionality with new implementations
  2. Branch by Abstraction: Create abstractions that allow both old and new implementations to coexist
  3. Parallel Implementation: Build the new system alongside the old one, then switch over
Success Metrics:
  • Comprehensive test coverage
  • No regressions in functionality
  • Improved code metrics (complexity, etc.)
  • Clear documentation of changes
  • Team understanding of the new implementation

Commit Best Practices

Effective commit practices are essential for maintaining a clear project history and facilitating team collaboration. This section covers best practices for creating meaningful commits and managing your changes.

Effective vs. Ineffective Commits

Ineffective Commits

git add .
git commit -m "fixes"
  • Uses git add . (adds everything)
  • Vague commit message without context
  • No reference to tickets or issues
  • Potentially includes unrelated changes
  • Could contain temporary files or logs
Bad Message Examples:
"Fixed stuff"
"WIP"
"Updates"
"Changed login.js"
"asdf1234"

Effective Commits

git add src/components/LoginForm.js
git add src/services/auth.js
git commit -m "Fix authentication timeout issue (#253)"
  • Adds specific files related to the change
  • Clear, descriptive commit message
  • References the issue number
  • Contains only relevant changes
  • Focused on a single logical change
Good Message Examples:
"Add password reset functionality (#142)"
"Fix data loading error in dashboard charts"
"Improve form validation accessibility"
"Refactor user authentication service for better testability"
"Update dependencies to fix security vulnerabilities"

Commit Message Structure

Short summary (50 chars or less)

More detailed explanatory text, if necessary. Wrap it to about 72
characters. The blank line separating the summary from the body is
critical.

Explain the problem that this commit is solving. Focus on why you
are making this change as opposed to how (the code explains that).
Are there side effects or other unintuitive consequences of this
change? Here's the place to explain them.

Further paragraphs come after blank lines.

- Bullet points are okay, too
- Typically a hyphen or asterisk is used for the bullet, preceded
  by a single space

Resolves: #123
See also: #456, #789

Components:

  • Subject Line: Brief summary (50 chars max)
  • Body: Detailed explanation if needed
  • Problem Statement: Why this change is needed
  • Issue References: Links to related tickets
Using Conventional Commits:
feat: add user authentication
fix: resolve login timeout issue (#123)
docs: update API documentation
style: format code according to style guide
refactor: simplify payment processing logic
test: add unit tests for auth service
chore: update dependencies

Best Practices:

  • Use the imperative mood ("Add feature" not "Added feature")
  • Capitalize the subject line
  • Do not end the subject line with a period
  • Separate subject from body with a blank line
  • Use the body to explain what and why vs. how
  • Reference issues and pull requests liberally
Multi-line Commit Example:
Implement password reset functionality

- Add password reset form component
- Create email notification service
- Connect to authentication API
- Add unit tests for reset flow

Resolves: #142

The Value of Specific File Additions

Why Avoid git add .?

  • Prevents accidental inclusion of:
    • Temporary files (.tmp, .swp)
    • Log files
    • Local configuration changes
    • Build artifacts
    • Environment-specific files
  • Forces conscious review of each changed file
  • Makes you think about logical grouping of changes
  • Helps maintain atomic commits
  • Prevents unrelated changes in single commits

Better Alternatives:

Use specific file paths:
git add src/components/LoginForm.js
Add files interactively:
git add -p

Lets you review and select specific chunks of changes

Add by pattern (carefully):
git add src/components/*.js

Adds all JavaScript files in a specific directory

Use Visual Tools:

Use Git GUI tools like VS Code's source control panel to selectively stage changes

Commit Organization Strategies

  • Atomic Commits: Each commit should address a single logical change
  • Frequent Small Commits: Commit often to create a detailed history
  • Logical Grouping: Group related changes in a single commit
  • Separate Concerns: Keep functional changes separate from style/refactoring changes
  • Complete Changes: Ensure each commit leaves the codebase in a functioning state

Useful Git Commands

View changes before committing:

git diff

View staged changes:

git diff --staged

Unstage a file:

git restore --staged <file>

Amend the previous commit:

git commit --amend

View commit history for a file:

git log --follow <file>
